# 合同执行情况跟踪共享平台 — 产品需求文档 (PRD)

## 1. 项目信息

| 字段 | 内容 |
|------|------|
| **项目名称** | contract_tracker |
| **编程语言 / 技术栈** | Vite + React + MUI + Tailwind CSS（默认） |
| **原始需求** | 构建一个公司内部合同执行情况跟踪共享平台，支持业务员通过外网访问，对合同信息、开票信息和收款跟踪进行统一的录入、存储、修改和查看。 |

---

## 2. 产品定义

### 2.1 产品目标

1. **提升合同执行透明度**：让业务员和管理者能实时掌握每份合同的签约、开票、回款全链路状态，减少信息断层。
2. **降低手工对账成本**：通过合同-开票-收款的关联结构，自动汇聚数据，替代零散的 Excel/纸质记录。
3. **保障数据安全与归属**：基于业务员登录体系，确保每个人只能查看和维护与自己相关的合同数据，同时支持外网安全访问。

### 2.2 用户故事

1. **As a** 业务员，**I want** 通过账号密码登录系统，**so that** 我只能看到和维护自己负责的合同信息，避免数据泄露和误操作。
2. **As a** 业务员，**I want** 录入和修改合同信息（编号、名称、签约方、金额、日期、状态等），**so that** 合同台账能随业务进展实时更新。
3. **As a** 业务员，**I want** 为每份合同添加开票记录（发票号、开票日期、金额、类型），**so that** 开票进度可追溯。
4. **As a** 业务员，**I want** 针对每张开票记录跟踪实际收款情况（收款日期、金额、回款状态），**so that** 回款逾期能及时预警。
5. **As a** 管理层 / 管理员，**I want** 从外网访问平台查看所有业务的合同执行概览，**so that** 无需内网即可掌握回款进度。

## 3. 技术规范

### 3.1 需求池 (Requirements Pool)

#### P0（必须有）
| 需求 | 说明 |
|------|------|
| 业务员登录 / 登出 | 账号+密码认证，会话管理 |
| 合同信息 CRUD | 合同编号、合同名称、签约方、合同金额、签订日期、合同状态的新增/编辑/删除/查询 |
| 开票记录管理 | 每份合同下关联多张开票记录（发票号、开票日期、开票金额、开票类型），支持增删改查 |
| 收款跟踪管理 | 每张开票记录关联收款信息（收款日期、收款金额、回款状态），支持增删改查 |
| 数据权限隔离 | 业务员只能查看和维护自己名下的合同数据 |
| 外网访问部署 | 支持 NAT 穿透方案（如花生壳），通过域名或公网 IP 访问 |

#### P1（应该有）
| 需求 | 说明 |
|------|------|
| 合同列表页可视化 | 列表展示所有合同，支持按状态/签约方/时间筛选和搜索 |
| 合同详情页 | 一张页面展示合同基本信息 + 关联开票记录 + 收款状态汇总 |
| 回款逾期提示 | 开票后超过约定天数未收款时，在列表中标记为"逾期" |
| 数据导出 | 合同台账 / 开票记录支持导出为 Excel |

#### P2（最好有）
| 需求 | 说明 |
|------|------|
| 仪表盘 | 回款率、逾期率等关键指标的统计看板 |
| 操作日志 | 记录谁在什么时间修改了什么数据 |
| 审批流 | 合同状态变更（如"已签约→已完结"）需提交审核 |
| 移动端适配 | 页面在手机浏览器上基本可用 |

### 3.2 UI 设计稿（文字描述）

#### 页面结构

1. **登录页 (`/login`)**
   - 中心区域为登录卡片：用户名输入框 + 密码输入框 + 登录按钮
   - 底部显示系统名称 "合同执行跟踪平台"

2. **合同列表页 (`/contracts`)**
   - 顶部导航栏：平台 Logo + 当前用户名 + 退出登录按钮
   - 筛选区：合同状态下拉（全部/已签约/执行中/已完结）、签约方搜索框、日期范围选择器
   - 操作按钮："新增合同" 按钮（右上角）
   - 主区域：表格列表，列包括——合同编号、合同名称、签约方、合同金额、签订日期、状态、回款进度（百分比）、操作（查看/编辑/删除）
   - 支持分页

3. **合同新增/编辑页 (`/contracts/new`、`/contracts/:id/edit`)**
   - 表单布局：合同编号（自动生成或手动输入）、合同名称、签约方、合同金额、签订日期（日期选择器）、合同状态（下拉选择：已签约/执行中/已完结）
   - 底部：保存 / 取消 按钮

4. **合同详情页 (`/contracts/:id`)**
   - **合同基本信息卡片**：展示合同关键字段，右侧提供"编辑"按钮
   - **开票记录区块**：以子表格形式展示该合同下的所有开票记录
     - 列：发票号、开票日期、开票金额、开票类型（增值税专票/普票/收据）、操作（新增/编辑/删除）
   - **收款跟踪区块**：展开每张开票记录后，展示对应的收款信息
     - 列：收款日期、收款金额、回款状态（未回款/部分回款/已回款）、操作（编辑）
   - 每条开票记录旁以进度条或标签显示该发票的回款完成度

5. **权限导航**
   - 普通业务员：仅见自己的合同
   - 管理员（可选角色）：可查看全部合同

### 3.3 数据模型概要

```
合同 (Contract)
├── id, contract_no, contract_name, party, amount, signed_date, status, salesperson_id
│
├── 开票记录 (Invoice)
│   ├── id, contract_id, invoice_no, invoice_date, amount, invoice_type
│   │
│   └── 收款记录 (Payment)
│       └── id, invoice_id, payment_date, amount, status
```

### 3.4 待确认问题 (Open Questions)

1. **合同编号规则**：合同编号是系统自动生成（如 YYYYMMDD-NNNN），还是业务员手动输入？
2. **合同状态枚举**：初始状态有哪些（至少包含：已签约 / 执行中 / 已完结）？是否需要"已逾期"等状态？
3. **回款逾期天数**：逾期提醒的阈值天数是多少（以开票日起算，30天/60天/可配置）？
4. **管理员角色**：是否需要独立的"管理员"账号来管理所有业务员账号，还是初期由开发人员直接操作数据库维护？
5. **数据导出范围**：导出 Excel 时默认导出当前业务员的所有数据，还是允许自定义筛选导出？
6. **部署环境**：外网访问方案是否确定使用花生壳？是否需要 HTTPS（SSL 证书）支持？
